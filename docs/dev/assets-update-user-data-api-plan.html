assets.updateUserData 接口调整计划

PR #650 反馈补充 / PinK 资源编辑面板 / 2026-06-26

结论

拆分接口 updateUserData 做整体替换，updateUserDataByPath 做单字段修改。
一次保存 常规 PinK 面板提交使用完整 userData，只触发一次 meta 写入和一次资源重导。
批量后置 多选资源需要批量接口，但不阻塞下个版本常规资源编辑对接。
当前 path, value 形式不能表达“空 path 代表完整 userData 替换”，且容易误触发字段写入语义。最终方案是保留路径能力，但不让空 path 复用成整体更新。

背景与问题

主题现状影响处理结论
PinK 常规资源编辑 PinK 面板已持有完整 userData，编辑后提交整体对象。 逐字段调用会产生多次 meta 写入，父子资源联动时成本更高。 提供 updateUserData(uuid, userData)。
AI / 精确修改 AI 可能只知道某个配置路径和值。 整体替换会要求调用方先读完整 userData，增加上下文和误删风险。 新增 updateUserDataByPath(uuid, path, value)。
子资源 UUID PR #650 已支持 parentUuid@subMetaId。 需要继续兼容子资源 userData 写入。 两个接口均支持父资源 UUID 和子资源复合 UUID。
saveAssetMeta 整份 meta 保存能力天然能改父子 userData，但公开接口语义过宽。 调用方可能修改非 userData 字段，破坏 meta 结构。 不恢复公开 saveAssetMeta，只暴露 userData 专用接口。

接口方案

常规接口：整体替换

TypeScript
updateUserData(uuid: string, userData: Record<string, any>): Promise<Record<string, any>>

// MCP / public lib:
assets.updateAssetUserData(uuid, userData)
  • 写入目标是该资源对应 meta 的 userData。
  • 传入对象整体替换旧 userData，不存在字段残留。
  • 单次调用只执行一次 asset.save() 和一次 asset._assetDB.reimport(asset.uuid)。
  • 不接受数组、字符串、数字、布尔值、null。

精确接口：路径更新

TypeScript
updateUserDataByPath(uuid: string, path: string, value: any): Promise<Record<string, any>>

// MCP / public lib:
assets.updateAssetUserDataByPath(uuid, path, value)
  • path 使用 lodash dot path 语义，例如 texture.wrapMode。
  • path 必须非空；整体替换必须走 updateUserData。
  • 适用于 AI 或脚本只修改单个配置项的场景。

后续批量接口

批量接口不进入本次最小交付范围；常规整体更新先支撑 PinK 下个版本资源编辑功能。
Draft
updateUserDataBatch(items: Array<{
  uuid: string;
  userData: Record<string, any>;
}>): Promise<Array<{
  uuid: string;
  userData: Record<string, any>;
}>>
批量场景建议规则原因
同名 UUID 重复 按归一化 UUID 合并，后出现的完整 userData 覆盖先出现的完整 userData。 整体替换语义要避免字段残留；调用方通常不会传重复项。
父资源与子资源同时修改 按父 meta 文件聚合，先修改父 userData，再修改各 subMetas[id].userData，最后每个父 meta 只保存一次。 减少 meta 文件写入次数，并保留 PR #650 的子资源能力。
多选资源 按资源所属 meta 文件分组并串行保存。 避免多个调用同时写同一 meta 文件。
错误处理 任一资源不存在则返回明确失败；是否支持部分成功需另行确认。 PinK 面板需要确定性反馈，避免静默丢配置。
